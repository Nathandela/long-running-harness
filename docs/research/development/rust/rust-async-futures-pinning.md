# Async Rust: Futures, Pinning, and Zero-Cost Runtimes

*2026-03-21*

## Abstract

Asynchronous programming in Rust represents one of the most ambitious attempts in programming language design to provide user-space concurrency without sacrificing the zero-cost abstraction principle that defines the language. Where other languages either impose a runtime with garbage collection (Go, Java), accept heap allocation and dynamic dispatch as inherent costs (C#, Kotlin), or rely on a single-threaded event loop (Node.js), Rust compiles async/await syntax into deterministic state machines that require no heap allocation, no dynamic dispatch, and no runtime beyond what the programmer explicitly selects. This design enables async Rust programs to run on bare-metal embedded microcontrollers and in WebAssembly sandboxes with the same source-level abstractions used for high-throughput network servers.

The cost of this ambition is substantial complexity. The `Future` trait's poll-based model inverts control flow compared to the callback or continuation-passing approaches used elsewhere. The `Pin` type, introduced to solve the problem of self-referential state machines generated by the compiler, adds a layer of abstraction with no precedent in other mainstream languages. The absence of a built-in runtime means the ecosystem has fractured across multiple executor implementations with incompatible extension traits and spawn semantics. The stabilization of async functions in traits, a feature requested since 2018, was not completed until Rust 1.75 in December 2023, and fundamental features like async drop, async closures, and runtime-agnostic spawning remain partially resolved as of early 2026.

This survey provides a comprehensive analysis of Rust's async system from theoretical foundations through practical implementation. We examine the desugaring of async/await into state machines, the Pin and Unpin mechanism for memory-stable futures, the Waker-based notification architecture, the design space of executors and I/O drivers, the long path to async traits, and the open problems that continue to shape the language's evolution. We situate Rust's approach within the broader landscape of concurrency models including Go's goroutines, Java's virtual threads, Kotlin's coroutines, C#'s Task-based async, and Zig's colorless async I/O, drawing on over thirty sources spanning official documentation, language team RFCs, production case studies, and formal verification research.

## 1. Introduction

The problem of handling many concurrent I/O operations efficiently has driven decades of systems programming research. Dan Kegel's articulation of the C10K problem in 1999 formalized what web server developers had been encountering in practice: the thread-per-connection model, where each incoming connection is assigned a dedicated operating system thread, collapses under the weight of context-switching overhead, memory consumption from pre-allocated stacks, and scheduler contention when connection counts reach the tens of thousands [1]. The response across the industry was a shift toward event-driven architectures -- epitomized by nginx replacing Apache httpd, Node.js popularizing single-threaded event loops, and the development of operating system primitives like Linux's epoll (2002), BSD's kqueue (2000), and Windows IOCP (2001) that enable efficient multiplexing of I/O readiness notifications across thousands of file descriptors.

Rust entered this landscape in 2015 with version 1.0, having already removed its original green-threading runtime during the pre-1.0 period. The removal was motivated by three concerns that would prove foundational to the async design that followed: green threads required a runtime that imposed overhead even on programs that did not use concurrency, stack management for green threads conflicted with Rust's zero-cost abstraction principle, and the need for safe FFI with C libraries was incompatible with the stack-switching mechanics of green threads [2]. Go, by contrast, had the luxury of designing goroutines into the language from its inception in 2009, when the need for async abstractions was already clear and no existing synchronous ecosystem demanded backward compatibility [3].

The removal of green threads left a gap that the Rust community filled through a series of increasingly sophisticated library designs. Alex Crichton and Aaron Turon published the zero-cost futures design in August 2016, establishing the poll-based `Future` trait that would eventually be stabilized [4]. The futures 0.1 crate and the Tokio runtime emerged shortly thereafter, providing a practical but ergonomically rough experience that required extensive use of combinators and explicit state threading. The async/await syntax, stabilized in Rust 1.39 (November 2019), transformed the developer experience by allowing the compiler to generate state machines from sequential-looking code [5]. However, this was explicitly shipped as a "minimum viable product," and as withoutboats observed in late 2023, "the project has shipped almost no extensions to async/await in the four years since the MVP was released" [6].

The remainder of this paper is organized as follows. Section 2 establishes the theoretical foundations of async I/O and event-driven architectures. Section 3 provides a taxonomy of concurrency approaches across languages. Section 4 analyzes Rust's async mechanisms in detail across twelve subsections. Section 5 offers a comparative synthesis with trade-off analysis. Section 6 identifies open problems and active areas of research. Section 7 concludes.

## 2. Foundations

### 2.1 The Async Problem Space

The fundamental motivation for async programming is the mismatch between the rate at which I/O operations complete and the rate at which CPUs can execute instructions. A network round-trip takes on the order of milliseconds; a CPU can execute billions of instructions per second. In the thread-per-connection model, a thread that has issued a network read and is waiting for the response occupies a kernel thread that consumes memory (typically 2-8 MB of stack on Linux), contributes to scheduler overhead, and wastes CPU cycles during context switches. At ten thousand concurrent connections, this translates to tens of gigabytes of stack memory and thousands of context switches per scheduling quantum [1].

Event-driven architectures address this by multiplexing many logical operations onto a small number of OS threads -- ideally one per CPU core. The operating system's event notification facility (epoll on Linux, kqueue on macOS/BSD, IOCP on Windows) allows a single thread to monitor thousands of file descriptors for readiness, waking only when actual progress can be made [7]. This is the foundation of the reactor pattern: an event loop waits for I/O readiness notifications and dispatches callbacks or continuations to handle them.

The trade-off is programming model complexity. Callback-based event-driven programming, as epitomized by early Node.js, produces what is colloquially known as "callback hell" -- deeply nested, non-linear control flow where data and error propagation become difficult to follow [8]. The async/await syntax, pioneered by C# in 2012 and subsequently adopted by JavaScript (ES2017), Python (3.5), Kotlin, Swift, and Rust, addresses this by allowing programmers to write sequential-looking code that the compiler transforms into the state machines needed by the event loop.

### 2.2 Readiness-Based vs. Completion-Based I/O

A critical distinction in async I/O design is between readiness-based and completion-based models. In a readiness-based model, the program asks the operating system to notify it when a file descriptor is ready for an I/O operation, and then the program performs the operation itself. Linux's epoll and BSD's kqueue follow this model. In a completion-based model, the program submits an I/O operation to the operating system along with the buffers to use, and the OS notifies the program when the operation has completed. Windows IOCP and Linux's io_uring (introduced in kernel 5.1, 2019) follow this model [9].

The distinction has profound implications for API design. Readiness-based I/O is compatible with borrowed buffers: the program retains ownership of its buffers and merely reads into or writes from them when notified of readiness. Completion-based I/O requires the program to transfer buffer ownership to the kernel for the duration of the operation, since the kernel reads from or writes to those buffers asynchronously. This ownership transfer is fundamentally incompatible with Rust's borrow-based `Read` and `Write` traits, creating a tension that has driven the design of specialized runtimes like monoio and glommio [10].

### 2.3 Stackful vs. Stackless Coroutines

Concurrency abstractions can be classified by whether they allocate a separate call stack for each concurrent task. Stackful coroutines (green threads, fibers, goroutines) allocate a stack -- either a fixed-size stack or a segmented/growable stack -- for each coroutine, enabling suspension at any point in the call chain. Stackless coroutines compile each suspendable function into a state machine that stores only the local variables live across suspension points, requiring no separate stack allocation [11].

Go's goroutines are the canonical example of stackful coroutines. Each goroutine begins with a small stack (a few kilobytes) that grows as needed through stack copying -- a process that requires the runtime to identify and update all pointers to stack-allocated data, which is functionally equivalent to garbage collection [2]. Rust's pre-1.0 green threads used segmented stacks, which introduced unpredictable performance cliffs when hot loops triggered stack growth at segment boundaries [2].

Rust's async/await uses stackless coroutines. Each async function compiles to a state machine enum whose variants contain only the local variables that are live at each await point. The size of this state machine is known at compile time, enabling stack allocation of the entire future tree without heap allocation. This is the technical basis of the "zero-cost" claim: the generated code is comparable to what a programmer would write by hand using explicit state machines [4].

## 3. Taxonomy of Approaches

Concurrency models across programming languages vary along several axes: whether coroutines are stackful or stackless, whether an event loop or runtime is built into the language, whether async functions are syntactically distinguished from synchronous ones (the "function coloring" problem), and what guarantees the type system provides about concurrent safety.

**Stackless coroutines with explicit syntax (Rust, C#, JavaScript, Python, Kotlin, Swift).** These languages use async/await keywords that distinguish suspendable functions from synchronous ones. The compiler transforms async functions into state machines. The degree of zero-cost varies: Rust achieves true zero-cost through monomorphization and stack allocation of futures, while C# and Kotlin allocate coroutine frames on the heap and rely on garbage collection [12].

**Stackful coroutines with transparent syntax (Go, Java Virtual Threads).** Go's goroutines and Java's virtual threads (Project Loom, GA in Java 21) present the same API as ordinary function calls and thread spawning. The runtime manages scheduling transparently, eliminating function coloring at the cost of a mandatory runtime with garbage collection support. Java's virtual threads demonstrated the viability of this approach for JVM languages, achieving millions of concurrent virtual threads with dramatically lower resource consumption than platform threads [13].

**Colorless async with I/O parametrization (Zig).** Zig's evolving async design represents a novel approach where functions accept an I/O interface parameter that can be instantiated with different implementations -- from simple blocking I/O to sophisticated coroutine-based systems. This parametrization eliminates function coloring without requiring a heavyweight runtime, though the design has undergone significant revision and is not yet finalized [14].

**Single-threaded event loop (Node.js).** Node.js uses a single-threaded event loop powered by libuv, which abstracts platform-specific I/O multiplexing (epoll, kqueue, IOCP). All JavaScript executes on the main thread, with CPU-intensive work offloaded to a worker thread pool. This model is simple and eliminates data races by construction but cannot utilize multiple cores without explicit worker processes [15].

## 4. Analysis

### 4.1 The Future Trait and Poll Semantics

The `Future` trait is the foundation of Rust's async system. Its definition in `core::future` is:

```rust
pub trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}
```

The `poll` method attempts to drive the future toward completion. It returns `Poll::Ready(output)` when the future has produced its final value, or `Poll::Pending` when it cannot yet make progress. Critically, a future returning `Pending` must have arranged for the `Waker` (accessible through the `Context` parameter) to be called when progress becomes possible. Repeated polling without intervening wake notifications is called "busy-waiting" and defeats the purpose of async I/O [16].

This poll-based design was a deliberate departure from the continuation-passing style used in Rust's earlier futures 0.1 library and in most other languages' async implementations. In continuation-passing, a future accepts a callback that is invoked when the result is available; composition requires allocating and storing these callbacks, typically on the heap. The shift to polling was directly inspired by the success of Rust's `Iterator` trait, which uses the analogous `next(&mut self) -> Option<T>` method. As withoutboats observed, "The shift from continuations to polling is exactly the same shift that was performed with iterators in 2013" [2]. The polling model enables the compiler to compose multiple futures into a single state machine without intermediate allocations, achieving the same optimization benefits that iterator adaptors enjoy through monomorphization and inlining.

### 4.2 State Machine Transformation

When the Rust compiler encounters an async function, it generates an anonymous type that implements `Future`. This type is an enum whose variants correspond to the suspension points (await expressions) in the function, with each variant storing the local variables that are live at that point. Consider:

```rust
async fn example(x: u32) -> u32 {
    let a = compute(x).await;
    let b = transform(a).await;
    a + b
}
```

The compiler generates approximately the following structure:

```rust
enum ExampleFuture {
    Start { x: u32 },
    AwaitCompute { x: u32, fut: ComputeFuture },
    AwaitTransform { a: u32, fut: TransformFuture },
    Done,
}
```

The `poll` implementation is a match statement that dispatches on the current variant, polls the inner future if one exists, transitions to the next state on `Ready`, and returns `Pending` if the inner future is not yet complete. Assembly-level analysis confirms that the compiler generates jump table dispatch for state transitions, with the state variable stored at a fixed offset in the closure environment [17]. The inner futures are inlined into the parent state machine, eliminating the need for separate heap allocations for each level of the async call chain.

This transformation is the basis of the zero-cost claim. Aaron Turon's 2016 design document established the principle: "None of the future combinators impose any allocation. All function calls are statically dispatched, and there are no heap allocations" [4]. The generated state machine is comparable to hand-written C code that explicitly manages transitions through a state enum, but produced automatically from sequential-looking source code.

The size of the generated state machine is the maximum of its variants rather than the sum, since only one variant is active at any time. However, deeply nested async call chains can produce large state machines, and the total size can be surprising. This has motivated compiler improvements to minimize the size of captured state and to drop variables as soon as they are no longer live.

### 4.3 Pin and Unpin: Solving the Self-Referential Problem

The state machine transformation described above creates a fundamental problem when async functions contain references to local variables across await points. Consider:

```rust
async fn bar(x: i32, y: i32) -> i32 {
    let mut z = x + y;
    foo(&mut z).await;
    z
}
```

The compiled future must store both `z` and the future returned by `foo(&mut z)` in the same state variant. But the inner future holds a reference to `z`, which is a field of the same struct -- a self-reference. If the future is moved in memory (for example, by being placed into a `Box` or transferred between threads), the pointer to `z` becomes dangling [18].

The Rust language team considered and rejected two alternative solutions before arriving at `Pin`. Move constructors, borrowed from C++, would require runtime fixup of internal pointers after every move -- an overhead incompatible with zero-cost abstractions and infeasible when values might be stored in collections or passed through generic code. Offset-based pointers, where self-references store byte offsets rather than absolute addresses, fail because the compiler cannot distinguish at compile time between a self-referential pointer and an external reference [18].

A third approach, the `?Move` bound (analogous to `?Sized`), was explored extensively but abandoned because it was not backward-compatible. Adding a new `?Trait` to the language requires all existing code to either implicitly assume the bound or be updated -- and since fundamental operations like `DerefMut` use associated types, the change would have been pervasive [18].

`Pin<P>` takes a different approach entirely. Rather than marking types as immovable, it wraps a pointer to establish a "pinned typestate." A value behind `Pin<&mut T>` cannot be moved through safe code because `Pin` does not expose a `&mut T` (which would allow `mem::replace` or `mem::swap`). The guarantee is: from the moment a value is pinned, it must remain valid at the same address until its `drop` handler is called [19].

The `Unpin` auto trait provides the escape hatch that makes `Pin` practical. Most Rust types -- primitives, `String`, `Vec<T>`, and any struct composed of `Unpin` types -- automatically implement `Unpin`, meaning they can be freely extracted from `Pin` because they contain no self-references and are safe to move. For `Unpin` types, `Pin<&mut T>` behaves identically to `&mut T`. Only types that opt out of `Unpin` by including `PhantomPinned` or that are generated by the compiler (async state machines) require the full pinning discipline [19].

Pin projection -- obtaining a pinned reference to a field of a pinned struct -- is where the complexity concentrates. A type author must decide for each field whether pinning is structural (the field is pinned when the parent is) or non-structural (the field can be accessed via ordinary `&mut`). Structural pinning requires that the type's `Unpin` implementation depends on the field's `Unpin` status, and that the `Drop` implementation treats `self` as pinned. Getting this wrong is a soundness hole. The `pin-project` and `pin-project-lite` crates automate this process, generating safe projection methods through procedural macros [20]. The `pin-project` crate enforces that structurally pinned fields cannot be moved out in `Drop` implementations, closing a subtle soundness gap.

As withoutboats reflected, Pin "represents a complexity cliff" because it breaks the mental models that programmers build around ordinary references. The absence of language-level support for pin -- no implicit reborrowing, no special syntax for assignment, no compiler-generated projections -- makes it significantly more difficult to work with than standard references [18]. Proposals for pin ergonomics improvements, including autoreborrowing for pinned references, landed as experimental implementations in the compiler in 2025 [21].

### 4.4 The Waker Mechanism

The `Waker` type is the notification mechanism that connects futures to their executors. When a future returns `Poll::Pending`, it must arrange for the `Waker` (obtained from the `Context` parameter passed to `poll`) to be invoked when the future can make progress. The waker then schedules the task for re-polling by the executor [22].

The `Waker` type uses a virtual function table (vtable) rather than a trait object for dynamic dispatch. The `RawWaker` struct contains a data pointer and a reference to a `RawWakerVTable` with four function pointers: `clone`, `wake`, `wake_by_ref`, and `drop`. The data pointer is passed as the first parameter to each vtable function, serving as the `self` equivalent [22]. This design was chosen because `Waker` must be `Clone` (event sources need to store wakers for later invocation), but the `Clone` trait is not object-safe because its return type is `Self`. The raw vtable approach achieves dynamic dispatch of cloning without requiring trait object infrastructure.

Different executor implementations populate the vtable differently depending on their architecture. For embedded single-task executors, the waker might be implemented as a static reference to an `AtomicBool` flag. For multi-threaded executors with task queues, wakers are typically reference-counted pointers to task objects that are enqueued for execution when woken. The waker API is deliberately minimal to accommodate this range of implementations [23].

The `LocalWaker` type, added more recently, supports optimizations for single-threaded executors by removing the `Send` requirement. Executors that run all tasks on a single thread can avoid the overhead of atomic reference counting in their waker implementations by using `LocalWaker` instead [23].

A critical implementation detail is that the waker stored by a future must be updated on every poll, because a future may have been moved to a different task with a different waker between polls. The async book explicitly warns: "We have to update the `Waker` every time the future is polled because the future may have moved to a different task with a different `Waker`" [22].

### 4.5 The Reactor-Executor Split

Rust's async architecture is characterized by a clean separation between two components: the executor, which manages compute resources by scheduling and polling futures, and the reactor, which manages I/O resources by monitoring event sources and waking futures when I/O operations can proceed. Together, these form the event loop, but Rust's design allows them to be independently implemented and combined [24].

The executor maintains a queue of tasks (top-level futures) and polls each task when it is woken. A simple executor might use a single thread with a FIFO queue; a sophisticated one like Tokio's uses a multi-threaded work-stealing scheduler where idle threads steal tasks from busy threads' queues. The executor is responsible for creating `Waker` instances that, when invoked, place the associated task back on the run queue.

The reactor registers interest in I/O events with the operating system (via epoll, kqueue, or IOCP) and holds a mapping from event source identifiers to the corresponding `Waker` instances. When the OS signals that a file descriptor is ready, the reactor looks up the associated waker and calls `wake()`, which causes the executor to schedule the task for polling.

This separation is mediated by the `mio` (Metal I/O) crate, which provides a cross-platform abstraction over OS-specific event notification facilities. Mio presents a unified API for registering interest in I/O events and polling for readiness, internally handling the differences between epoll's level-triggered and edge-triggered modes, kqueue's filter-based interface, and IOCP's completion-based model [7]. The `polling` crate from the smol ecosystem provides a similar abstraction with broader platform support including illumos event ports and VxWorks poll [7].

### 4.6 Executors and Runtimes

Rust does not include an async runtime in its standard library, leaving this to the ecosystem. This was a deliberate design decision: a built-in runtime would either impose overhead on programs that do not use async (violating zero-cost) or would need to satisfy all possible use cases from embedded to high-throughput servers. The result is a diverse ecosystem of runtimes optimized for different scenarios.

**Tokio** is the dominant runtime, with over 437 million total downloads and 20,768 dependent crates as of early 2026 [25]. Tokio provides a multi-threaded work-stealing scheduler as its default mode, where a pool of worker threads cooperatively execute tasks. When one thread's local queue is exhausted, it steals tasks from other threads' queues, achieving load balancing without centralized scheduling. This design requires that spawned tasks be `Send + 'static`, as they may execute on any worker thread, which mandates the use of `Arc` and `Mutex` (or Tokio's own async-aware `Mutex`) for any shared state. Tokio also provides a single-threaded "current-thread" runtime for use cases where `Send` bounds are undesirable. Beyond the core executor, Tokio includes modules for file system operations, networking, process management, signal handling, and timers, making it more of a framework than a bare runtime [26].

**async-std** was introduced as a "standard library for async Rust" that mirrored the std API surface with async versions. However, fundamental design challenges -- including the decision to spawn threads for blocking I/O operations and incomplete platform support -- led to its official discontinuation on March 1, 2025. The recommended replacement is smol [26].

**smol** emphasizes simplicity and transparency, with approximately 1,000 lines of core executor code. It uses the `polling` crate for I/O and the `async-executor` crate for task scheduling, with each component independently usable. Smol's design philosophy prioritizes predictable performance, particularly tail latency, over raw throughput. Benchmarks have shown that smol uses approximately 15% less memory overhead than Tokio during concurrent operations [26]. A notable property is that smol-based code can run within a Tokio runtime (but not vice versa), making it a pragmatic choice for library authors seeking runtime independence.

**Embassy** is an async runtime designed specifically for embedded systems where multi-threading is infeasible and heap allocation is prohibited. Tasks are statically allocated at compile time; if they do not fit in RAM, the linker detects this as a compile-time error rather than a runtime panic. Embassy's executor automatically puts the CPU core to sleep when no tasks are ready, waking on interrupts. It requires no dynamic memory allocation and runs on a single stack, eliminating per-task stack size tuning. Embassy obsoletes the traditional RTOS with kernel context switching, achieving smaller code size and lower overhead [27]. The Embassy ecosystem includes `embassy-net` for TCP/UDP/IP networking, `embassy-usb` for device-side USB, and hardware abstraction layers for STM32, nRF, and RP2040 microcontrollers.

**Glommio** (from Datadog) implements a thread-per-core, shared-nothing architecture built entirely on Linux's io_uring. Unlike other runtimes, Glommio uses no helper threads; each CPU core operates with its own set of io_uring ring buffers and processes tasks independently. Specifically, each thread maintains three io_uring instances: a main ring for normal operations, a latency ring for time-sensitive operations, and a polling ring for monitoring. Glommio's scheduler implements proportional-share scheduling to ensure fairness between task classes. Direct I/O is a first-class citizen, with optional buffered I/O support [28].

**Monoio** (from ByteDance) is another io_uring-based, thread-per-core runtime. Unlike tokio-uring, which runs atop a standard Tokio runtime, monoio is a standalone runtime that communicates directly with io_uring (or epoll/kqueue as fallback). A key design distinction is buffer ownership transfer: because io_uring requires the kernel to have direct access to user-space buffers during operations, monoio uses a "rent" model where the caller transfers buffer ownership to the runtime, which returns it after the operation completes [10].

### 4.7 I/O Models and io_uring Integration

The traditional async I/O stack in Rust is readiness-based: Tokio uses mio, which wraps epoll/kqueue/IOCP, to monitor file descriptors for readiness. When a socket is ready for reading, the runtime polls the corresponding future, which performs the actual `read()` syscall. This model is well-understood and compatible with Rust's borrowing semantics, since the program retains ownership of its buffers throughout.

Linux's io_uring, introduced in kernel 5.1 (2019) by Jens Axboe, represents a paradigm shift to completion-based I/O. The application and kernel share two ring buffers: the Submission Queue (SQ), where the application enqueues I/O requests, and the Completion Queue (CQ), where the kernel places results [9]. This design minimizes syscalls -- multiple operations can be submitted in a single `io_uring_enter()` call -- and enables the kernel to optimize operations internally, including request merging and kernel-side polling that eliminates syscalls entirely for high-throughput workloads.

The integration of io_uring with Rust's async model presents both opportunities and challenges. The Apache Iggy project's migration from Tokio to a thread-per-core io_uring architecture (completed February 2026) provides an instructive case study. Their benchmarks showed that while raw throughput was comparable between the two architectures, tail latencies improved dramatically: P95 latency improved 28%, P99 improved 32%, and P9999 improved 92% (from 86.3ms to 7.17ms) [29]. The migration revealed several architectural challenges:

First, Rust's `Future::poll` model must be adapted for completion semantics. The initial poll submits the operation to the SQ; subsequent polls check the CQ for completion using opaque cookies to match submissions with completions [29]. Second, `join!()` does not guarantee operation ordering, which can corrupt block device state when operations must be sequenced. The io_uring `IOSQE_IO_LINK` flag provides operation chaining, but this capability is not exposed through current Rust async runtime abstractions [29]. Third, current runtime abstractions (File, TcpListener, TcpStream) expose only POSIX-compatible io_uring features, leaving capabilities like request chaining, registered buffers, and oneshot APIs inaccessible without low-level ring access.

The Iggy team evaluated three runtimes -- monoio (deemed unmaintained), glommio (also essentially unmaintained), and compio -- ultimately selecting compio for its active development, broad io_uring feature coverage, and decoupled driver-executor architecture [29].

### 4.8 Async Traits: The Long Stabilization Saga

The inability to write `async fn` in trait definitions was one of the most persistent pain points in Rust's async ecosystem. The fundamental difficulty is that an async function returns an anonymous future type whose size depends on the function body. In a trait, the return type of an async method must be expressible as an associated type, but each implementor's future has a different size and layout. This is the same problem that `-> impl Trait` in traits poses, and the solution -- Return Position Impl Trait In Traits (RPITIT) -- required years of compiler work to implement [30].

Before stabilization, the community relied on the `#[async_trait]` procedural macro by David Tolnay, which desugars async trait methods into methods returning `Box<dyn Future + Send>`. This works but imposes heap allocation and dynamic dispatch on every async method call, negating the zero-cost property that motivates Rust's async design [30].

RPITIT and async fn in traits were stabilized in Rust 1.75 (December 2023). Each implementor's async method returns its own concrete future type, which the compiler infers as an anonymous associated type. This preserves zero-cost: no boxing, no dynamic dispatch, full monomorphization [30].

However, stabilization came with significant limitations. First, traits using RPITIT are not dyn-compatible (not object-safe), meaning they cannot be used with trait objects for dynamic dispatch. This prevents patterns like `Box<dyn MyAsyncTrait>` that are common in plugin architectures and dependency injection [30].

Second, there is no way to add `Send` bounds to the returned future in generic code. If a library defines:

```rust
trait HttpService {
    async fn fetch(&self, url: Url) -> HtmlBody;
}
```

then a consumer who needs to spawn the future on a multi-threaded executor cannot write `where T: HttpService, T::fetch(..): Send` because the return type is anonymous and unnameable. This is the "Send bounds problem" that blocks adoption of native async traits in core ecosystem crates like Tower and Hyper [31].

The `trait_variant` crate, released alongside the stabilization, provides a workaround through a procedural macro that generates parallel trait definitions with and without Send bounds:

```rust
#[trait_variant::make(HttpService: Send)]
pub trait LocalHttpService {
    async fn fetch(&self, url: Url) -> HtmlBody;
}
```

This generates both `LocalHttpService` (for single-threaded use) and `HttpService` (with Send-bounded futures, for multi-threaded executors) [30]. Return Type Notation (RTN), proposed in RFC #3654, would provide a more direct solution by allowing bounds like `T::fetch(..): Send` in where clauses, but this remains unstabilized as of early 2026 [6].

The `dynosaur` proc macro addresses the dyn-compatibility gap by generating wrapper types that heap-allocate the future when used through trait objects, recovering dynamic dispatch at the cost of allocation [21].

### 4.9 Structured Concurrency and Cancellation

Structured concurrency is the principle that concurrent operations should be scoped to a lexical block, with all spawned tasks completing (or being cancelled) before the block exits. This principle, articulated by Martin Sústrik and popularized by Nathaniel J. Smith's "Notes on structured concurrency," has been adopted as a first-class feature in Kotlin (coroutine scopes) and Swift (task groups) [32].

Rust's async ecosystem provides several mechanisms that approximate structured concurrency without enforcing it as a language-level invariant:

**`tokio::select!`** races multiple futures concurrently, completing when the first branch resolves and dropping (cancelling) the remaining branches. This is the most common concurrency primitive in Tokio-based code, but it introduces the cancellation safety problem: dropped futures lose any intermediate state accumulated since their last yield point [33].

**`tokio::task::JoinSet`** manages a dynamic collection of spawned tasks, providing methods to await their completion and collect results. Unlike raw `tokio::spawn`, `JoinSet` provides a handle for awaiting all tasks and propagating errors, approaching structured concurrency semantics.

**Cancellation semantics** in Rust differ fundamentally from most other languages. Cancellation is synchronous: dropping a future immediately and synchronously releases its resources through the standard `Drop` trait. There is no asynchronous cancellation callback, no grace period, and no cooperative cancellation protocol built into the language. This is both a strength (cancellation is deterministic and zero-cost) and a weakness (resources that require async cleanup, such as open database transactions, cannot be properly finalized during cancellation) [33].

A future is considered "cancellation-safe" if dropping it before completion has no observable side effects beyond not producing a result. The Oxide Computer Company's RFD 400 provides the most thorough treatment of cancellation safety in production async Rust, distinguishing between cancel safety (a local property of individual futures) and cancel correctness (a global property of the system). The document identifies three conditions for cancel correctness violations: the system contains cancel-unsafe futures, those futures are actually cancelled, and the cancellation violates system invariants [33]. Practical recommendations include the reserve pattern (splitting operations into cancel-safe acquisition and potentially-unsafe execution phases), creating futures outside `select!` loops and resuming them via pinned mutable references, and using explicit cancellation channels instead of task aborts.

In comparison, Kotlin enforces structured concurrency through its Job hierarchy, where child coroutines cannot outlive their parents, and cancellation propagates from parent to child. Swift's task groups provide similar guarantees. Rust lacks this enforcement at the language level, relying instead on library conventions and programmer discipline [32].

### 4.10 Streams and Async Iteration

The `Stream` trait (also known as `AsyncIterator`) is the async analog of `Iterator`, producing a sequence of values over time. Its definition follows the same poll-based pattern:

```rust
pub trait AsyncIterator {
    type Item;
    fn poll_next(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Option<Self::Item>>;
}
```

The naming has undergone revision. The trait was originally named `Stream` in the futures crate, renamed to `AsyncIterator` when moved into the standard library, and a 2024 proposal by Yoshua Wuyts suggested renaming it back to `Stream` while introducing a new AFIT-based `AsyncIterator` that uses `async fn next(&mut self) -> Option<Self::Item>` instead of the poll-based interface [34].

Async generators, using `async gen fn` or `async gen {}` block syntax, would allow the natural expression of asynchronous iteration:

```rust
async gen fn read_lines(path: &Path) -> io::Result<String> {
    let file = File::open(path).await?;
    let reader = BufReader::new(file);
    for await line in reader.lines() {
        yield line?;
    }
}
```

The `for await` loop consumes async iterators, providing the consumption counterpart to `yield` in async generators. Together, `async gen {}` provides the context, `yield` provides the iteration effect, `await` provides forward asynchrony, and `for await` completes the iteration [34]. As of mid-2025, an experimental implementation of a builtin `iter!` macro for unpinned generators was landed in the compiler, and the language team held productive discussions about the generator trait design, but full stabilization of async generators remains pending [21].

### 4.11 The Function Coloring Problem

Bob Nystrom's 2015 essay "What Color is Your Function?" articulated a criticism that applies to most async/await designs: async functions and synchronous functions are two different "colors" that cannot freely call each other [35]. An async function can call synchronous functions, but a synchronous function cannot directly await an async function's result without blocking the current thread. This creates a "viral" effect where the use of async at any level of the call stack tends to propagate upward, requiring callers to also be async.

In Rust, the coloring manifests in several ways. The `Future` trait's `poll` method takes `Pin<&mut Self>`, which cannot be called from non-async code without an executor. Bridging from synchronous to async code requires `block_on()` (which blocks the current thread until the future completes) or spawning a runtime. Bridging from async to synchronous code that might block requires `spawn_blocking()` to avoid starving the executor's worker threads.

The defense of Rust's function coloring, articulated by Jimmy Hartzell and others, rests on two arguments [3]. First, Rust allows confining the coloring boundary to a precise location -- a `block_on()` call at the top of `main()` or at a service boundary -- rather than having it pervade the codebase. Second, the distinction between sync and async functions carries genuine semantic information: async functions can suspend, and the type system ensures that suspension is handled correctly through the entire call chain.

Go eliminates function coloring entirely through goroutines: every function can perform I/O, and the runtime transparently suspends goroutines that would block. Java's virtual threads (Project Loom) achieve the same transparency for JVM code. The trade-off is that these approaches require a runtime capable of stack switching and garbage collection, which Rust's design constraints preclude [13].

Zig's approach represents a middle path: functions are parametrized over their I/O interface, allowing the same code to execute synchronously or asynchronously depending on the provided interface implementation. This eliminates syntactic coloring while preserving the ability to use different execution strategies, though it introduces its own form of complexity through interface parametrization [14].

### 4.12 Runtime Agnosticism and Ecosystem Fragmentation

A persistent challenge in Rust's async ecosystem is the difficulty of writing runtime-agnostic library code. The core `Future` trait is defined in the standard library, but essential operations like spawning tasks, sleeping, setting timers, and performing I/O are provided by runtimes and have no standardized trait interface. This means that a library using `tokio::spawn()` is coupled to Tokio and cannot be used with smol or Embassy without conditional compilation [36].

The problem is structural. Task spawning requires a runtime-specific function because different runtimes have different task representations, scheduling strategies, and capability requirements (`Send` bounds differ between single-threaded and multi-threaded runtimes). Timer implementations depend on the runtime's time driver. I/O types (TcpStream, File) are runtime-specific because they integrate with the runtime's reactor.

Several approaches have been attempted. The `agnostic` crate provides a unified API for task spawning, networking, and DNS resolution that works with both Tokio and smol through feature flags [36]. The `agnostic-lite` variant is no_std-compatible and allocation-free. Library authors can also accept trait bounds for spawn functions, though this conflicts with Rust's orphan rules and adds complexity to public APIs. The most pragmatic approach is to write compute-only library code that accepts and returns `impl Future` without spawning tasks or performing I/O directly, delegating those concerns to the application layer.

The ecosystem fragmentation has concrete consequences. Crates built on Tokio's timer or networking types (the majority of the ecosystem) cannot be used with alternative runtimes. This network effect has solidified Tokio's dominance, creating what some describe as a de facto standard that the language team is reluctant to formalize in the standard library [26].

## 5. Comparative Synthesis

The following table summarizes the key design trade-offs across major async concurrency implementations:

| Dimension | Rust async/await | Go goroutines | Java Virtual Threads | Kotlin coroutines | C# async/await | Node.js | Zig async |
|---|---|---|---|---|---|---|---|
| **Coroutine type** | Stackless | Stackful | Stackful | Stackless | Stackless | Stackless (callbacks) | Parametric |
| **Function coloring** | Yes (async/sync) | No | No | Yes (suspend) | Yes (async/sync) | Yes (async/sync) | No |
| **Runtime required** | External (Tokio, etc.) | Built-in | Built-in (JVM) | Built-in (kotlinx.coroutines) | Built-in (.NET) | Built-in (libuv) | Parametric |
| **Heap allocation** | Zero-cost (none required) | Per-goroutine stack | Per-virtual-thread stack | Coroutine frame on heap | State machine on heap | Closures on heap | Configurable |
| **GC required** | No | Yes | Yes | Yes (JVM/JS) | Yes (.NET GC) | Yes (V8 GC) | No |
| **Cancellation** | Synchronous drop | Context-based cooperative | Thread interruption | Structured (Job hierarchy) | CancellationToken | None (manual) | N/A |
| **Structured concurrency** | Library (JoinSet, select!) | Library (errgroup) | Language (StructuredTaskScope) | Language (coroutineScope) | Library (Task.WhenAll) | None | N/A |
| **Embedded/no_std** | Yes (Embassy) | No | No | No (Kotlin/Native partial) | No | No | Yes |
| **io_uring support** | Yes (monoio, glommio, compio) | Experimental | No | No | No | Experimental (libuv) | Planned |
| **Async in traits** | Since 1.75 (limited) | N/A (no traits) | N/A (interfaces work) | Suspend functions in interfaces | Async methods in interfaces | N/A | N/A |
| **Send safety** | Compile-time (Send + Sync) | Runtime race detector | Runtime (synchronized) | Structured concurrency | Runtime (SynchronizationContext) | Single-threaded (N/A) | Compile-time |

Several cross-cutting observations emerge from this comparison:

**Zero-cost vs. transparency trade-off.** Rust and Zig achieve zero-cost async (no mandatory heap allocation, no GC) at the cost of function coloring and explicit runtime management. Go and Java achieve transparent concurrency (no function coloring) at the cost of a mandatory runtime with garbage collection. C# and Kotlin occupy a middle ground with function coloring but GC-managed coroutine frames.

**Ecosystem maturity.** Despite Rust's async system being explicitly shipped as an MVP, the Tokio ecosystem has achieved remarkable production maturity. Cloudflare, Discord, AWS (Firecracker), and Dropbox have deployed Tokio-based services handling millions of concurrent connections. However, the ecosystem remains less mature than Go's or Java's for common patterns like structured concurrency, retry logic, and graceful shutdown.

**The embedded advantage.** Rust's async system is unique in supporting bare-metal embedded targets. Embassy demonstrates that the same async/await syntax used for server development compiles to efficient interrupt-driven code on microcontrollers with kilobytes of RAM, with statically allocated tasks and no heap. No other language in this comparison achieves this generality.

**I/O model evolution.** The shift from readiness-based to completion-based I/O (io_uring) is primarily a Rust and systems-level concern. Go's runtime abstracts I/O completely, and JVM-based languages rely on NIO. Rust's direct exposure of the I/O model creates both an opportunity (maximum performance through io_uring) and a fragmentation risk (readiness-based and completion-based APIs are fundamentally incompatible).

## 6. Open Problems & Gaps

### 6.1 Async Drop

The inability to run asynchronous code during resource cleanup is one of the most consequential gaps in Rust's async system. The standard `Drop` trait's `drop(&mut self)` method is synchronous; it cannot `.await` anything. This means that resources requiring asynchronous cleanup -- database connections that should send a graceful disconnect, file handles that should flush buffered writes through io_uring, network sessions that should send a shutdown frame -- cannot be properly finalized when a future is dropped [6].

The problem is not merely ergonomic. In completion-based I/O, an in-flight operation cannot be synchronously cancelled; the kernel completes it asynchronously. If a future holding an io_uring submission is dropped, the submitted operation continues but the buffer it references may be deallocated, leading to use-after-free in kernel space. Current workarounds involve leaking resources or blocking the thread during drop, neither of which is acceptable.

An early-stage experiment to support async drop was landed in the Rust compiler in 2025, but the design faces three fundamental challenges [21]. First, async drop must compose with synchronous drop: calling non-async code that drops an async-droppable value must still work, potentially by spawning the async drop on the ambient executor. Second, async drop must handle the case where the drop itself is cancelled (double drop). Third, the interaction with unwinding during panic is subtle: an async destructor that is interrupted by a panic must not leave resources in an inconsistent state.

### 6.2 Async Closures

Async closures -- closures that can `.await` inside their body -- were a critical gap that was partially addressed in the Rust 2024 edition [21]. Before stabilization, the workaround of writing `|x| async move { ... }` produced a closure that returns a future, but this future could not borrow from the closure's captures because the future's lifetime was independent of the closure's. True async closures, written as `async |x| { ... }`, produce futures whose lifetime is tied to the closure, enabling borrowing from captures.

The stabilized design models async closures using the existing `Fn`, `FnMut`, and `FnOnce` traits with `impl Future` return types, avoiding the creation of separate `AsyncFn` trait families. A remaining open question is whether async closures should return `impl Future` or `impl IntoFuture` -- a distinction that affects how the returned future's Send-ness interacts with the closure's captures [21].

### 6.3 Generator-Based Iteration

The `iter!` macro and `gen fn` syntax for synchronous generators, and their async counterparts `async gen fn`, are in active development. The fundamental design question is the generator trait: should generators use a poll-based interface (like `Future` and `Stream`), or should they use a simpler `next`-based interface that hides the polling? The poll-based approach is more efficient (it avoids an extra layer of state machine transformation) but harder to implement manually. The experimental `iter!` macro uses unpinned generators, which simplifies the implementation but prevents generators from containing self-references across yield points [21].

### 6.4 Keyword Generics and Effect Systems

The keyword generics initiative proposes enabling code that is generic over effects like `async`, `const`, `try`, and `unsafe`. The motivation is the combinatorial explosion of API surfaces: with five effects, the `Fn` trait family would require 96 variants instead of 6 to cover all combinations [37]. The proposed syntax uses `#[maybe(async)]` or `?async` annotations to indicate that a function works in both sync and async contexts.

The initiative has generated substantial controversy. Proponents argue that effect generics are essential for library authors who must currently duplicate entire API surfaces for sync and async variants (as seen in `reqwest` vs. `reqwest::blocking`). Critics argue that the complexity is prohibitive, that the proposed syntax is confusing, and that the interaction with Rust's type system creates unsolvable inference problems. As of early 2026, the initiative remains in the exploration phase without a concrete stabilization timeline [37].

### 6.5 Runtime-Agnostic Async in the Standard Library

The question of whether and how the standard library should provide async primitives beyond the `Future` trait remains unresolved. Candidates for standardization include the `AsyncIterator`/`Stream` trait, an async `Read`/`Write` trait, and potentially a minimal executor interface. Each raises difficult design questions: should async I/O traits use readiness-based or completion-based semantics? Should they require `Pin`? Should they be compatible with no_std? The async working group's reorganization into permanent subteams under the lang and libs committees, initiated in 2024, aims to provide the governance structure needed to make these decisions [31].

### 6.6 Formal Verification

The RustBelt project (PLDI 2018) proved type soundness for a core subset of Rust using the Iris concurrent separation logic, but this proof covers only the synchronous fragment of the language [38]. Extending formal verification to async Rust -- including the `Future` trait, `Pin`, `Waker`, and executor interactions -- remains an open research challenge. The complexity of the async state machine transformation, combined with the subtle invariants of `Pin` and the dynamic dispatch in `Waker`, makes formal verification significantly more challenging than for synchronous Rust.

## 7. Conclusion

Rust's async/await system represents a distinctive point in the design space of concurrent programming: it is the only production language that provides user-space concurrency through stackless coroutines with zero mandatory heap allocation, no garbage collector, and compile-time thread-safety guarantees, while supporting targets ranging from microcontrollers with kilobytes of RAM to servers handling millions of concurrent connections. This generality is achieved through a layered architecture where the language provides minimal primitives (`Future`, `Pin`, `Waker`, `Context`) and the ecosystem provides executors and I/O drivers tailored to specific deployment contexts.

The costs of this design are real and well-documented. The `Pin` type introduces a complexity cliff that breaks the mental models programmers build from working with ordinary references. The absence of a standard runtime fragments the ecosystem and makes runtime-agnostic library development difficult. The function coloring problem creates friction at async/sync boundaries. Cancellation safety requires careful reasoning that the type system does not enforce. Async fn in traits, while stabilized, lacks Send bounds and dyn-compatibility, limiting its applicability in production frameworks.

The trajectory of development since the 2019 MVP has been one of steady but slow progress on closing the gap between async and synchronous Rust. Async closures shipped in the 2024 edition. Pin ergonomics improvements and async drop experiments landed in the compiler in 2025. Generator syntax, async iteration, and Return Type Notation are in active development. The reorganization of the async working group into permanent subteams signals institutional commitment to sustained investment.

The comparative analysis reveals that each concurrency model involves fundamental trade-offs. Go and Java trade zero-cost abstraction for developer ergonomics and the absence of function coloring. Rust trades ergonomics for zero-cost abstraction and embedded compatibility. Zig's parametric approach eliminates coloring without a runtime but introduces its own complexity. The optimal choice depends on the deployment context: Rust's approach is unmatched for embedded, WebAssembly, and latency-critical server workloads where runtime overhead is unacceptable, while Go and Java remain more productive for typical cloud services where milliseconds of GC latency are tolerable.

The open problems -- async drop, runtime-agnostic APIs, formal verification, and the keyword generics question -- will define the next phase of Rust's async evolution. Their resolution will determine whether async Rust can achieve the same "just works" quality that characterizes synchronous Rust, or whether it will remain a powerful but sharp tool that demands expertise beyond what most developers possess.
